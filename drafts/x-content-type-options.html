Abstract

Often HTTP responses are delivered with a wrong or even missing Content-Type. In order to be compatible with these servers, User Agents employ content sniffing of the HTTP response, this can be a security hole, because untrusted content might be executed or displayed. X-Content-Type-Options: nosniff can be used to forcefully disable sniffing in the User Agent.

1 Introduction

The HTTP X-Content-Type-Options header field indiciates wether the User Agent should do mimetype sniffing for the requested resource. The only actionable value for the header field is nosniff at the moment, which should disable certain mime sniffing. This means when the Content-Type header field is specified, the user agent should try to render or execute the content according to this type, even when the operation is bound to produce failures. When the header field doesn't exists the User Agents should procced as if the Content-Type was set to text/plain. This is done for compatibility reasons.
